This continuation config defines the nitpick_fix_agent job. It follows the exact same structure as report-code-coverage.yml and desktop-app-release.yml.

Full content:

```yaml
version: 2.1

parameters:
  nitpick_issue_id:
    type: string
    default: ''

jobs:
  nitpick_fix_agent:
    docker:
      - image: cimg/node:22.22
    resource_class: medium
    environment:
      CLAUDE_CODE_USE_BEDROCK: "1"
      AWS_REGION: us-east-1
      PRE_COMMIT_AUTO_RUN: "1"
      ANTHROPIC_DEFAULT_SONNET_MODEL: "us.anthropic.claude-sonnet-4-6"
    steps:
      - checkout
      - run:
          name: Configure Git
          command: |
            git config --global user.name 'devrev-ci-bot'
            git config --global user.email 'ci-bot@devrev.ai'
      - run:
          name: Adding .npmrc file
          command: |
            echo -e "@devrev-private:registry=https://registry.npmjs.org\n//registry.npmjs.org/:_authToken=${NPM_PRIVATE_ACCESS_TOKEN}" > .npmrc
      - run:
          name: Install dependencies
          command: |
            npm install -g @anthropic-ai/claude-code
            npx pnpm@10.32.1 install --frozen-lockfile
      - run:
          name: Query nitpick issues
          command: |
            export NITPICK_ISSUE_ID="<< pipeline.parameters.nitpick_issue_id >>"
            if [ -z "${NITPICK_ISSUE_ID}" ]; then
              echo "Querying DevRev for pending nitpick issues..."
              ISSUES=$(bash scripts/circleci/query_nitpick_issues.sh)
              if [ -z "${ISSUES}" ]; then
                echo "No issues to process. Exiting."
                circleci-agent step halt
              fi
              # Process first issue only (one issue per job for concurrency safety)
              export NITPICK_ISSUE_ID=$(echo "${ISSUES}" | head -1)
            fi
            echo "export NITPICK_ISSUE_ID=${NITPICK_ISSUE_ID}" >> "${BASH_ENV}"
      - run:
          name: Run nitpick fix agent
          command: |
            bash scripts/circleci/run_nitpick_fix_agent.sh
          no_output_timeout: 30m

workflows:
  nitpick_fix:
    jobs:
      - nitpick_fix_agent:
          context:
            - github
            - aws-dev-account
            - npm
            - devrev-api
```

Key design decisions:
- `resource_class: medium` -- agent is network-bound (Bedrock API), not CPU-bound
- `no_output_timeout: 30m` -- agent may be silent during long analysis phases
- `circleci-agent step halt` -- graceful exit when no issues to process (does not fail the job)
- Single issue per job execution -- prevents concurrency conflicts (CICD-02)
- `BASH_ENV` used to pass NITPICK_ISSUE_ID between steps
- Contexts: `github` (gh CLI), `aws-dev-account` (Bedrock), `npm` (npmrc token), `devrev-api` (DEVREV_PAT)
- Claude Code CLI installed in the "Install dependencies" step alongside pnpm
- Git configured before any git operations (matching desktop-app-release.yml pattern)

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CircleCI scheduled pipeline -> Agent job | Cron trigger sends pipeline parameters; untrusted if API is publicly accessible |
| CircleCI API -> Manual dispatch | External caller can trigger pipeline with arbitrary nitpick_issue_id |
| Agent job -> AWS Bedrock | AWS credentials flow from CircleCI context to Bedrock API |
| Agent job -> GitHub | GitHub token from context enables branch push and PR creation |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-10 | Spoofing | Manual pipeline dispatch | mitigate | CircleCI API requires project-level Circle-Token for pipeline trigger; only team members with CircleCI access can dispatch; the agent validates the issue ID exists in DevRev before proceeding |
| T-04-11 | Denial of Service | Cron schedule + unbounded issue count | mitigate | Agent processes only 1 issue per job execution (head -1); if 10 issues queued, 10 separate jobs trigger over 10 hours; CircleCI resource class is medium (not consuming large runners) |
| T-04-12 | Tampering | CircleCI context secrets | accept | Contexts are managed by CircleCI's access control; only project members can view/modify; standard CircleCI security model |
| T-04-13 | Elevation of Privilege | Agent job has github + aws-dev-account contexts | mitigate | --allowedTools restricts what the agent can do with these credentials; post-run guardrails validate output; draft PRs require human approval before merge |
</threat_model>
